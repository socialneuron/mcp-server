# Full-server adversarial audit — 2026-07-21

Scope: the entire public MCP server at merged-main `d06d661` (post-#307), reviewed as four parallel adversarial passes — **SSRF/URL**, **auth/scope/OAuth**, **injection/scanner/secrets**, and **rate-limit/DoS/side-effects**. Every finding below was reproduced against the actual code with a concrete `file:line` and exploit path; unreproducible or backend-delegated claims were dropped or labelled as such.

Threat model: an attacker who can (a) relay untrusted web content through any tool input, (b) hold a low-tier API key or register an OAuth client, and (c) control a drifting/compromised backend response. Identity is always the caller's own token — `callEdgeFunction` signs every upstream call with the caller's key and injects the caller's `userId` (`edge-function.ts:143-170`), so **no cross-tenant credit/deletion path is reachable from this layer**; row ownership, credit deduction, and RLS live in the out-of-repo edge functions.

## Fixed in this branch (verified, tests added)

### SSRF-1 — IPv4-mapped / NAT64 IPv6 literals bypassed the SSRF guard · HIGH
`validateUrlForSSRF` blocked private IPv6 only via dotted-decimal regexes (`ssrf.ts` `BLOCKED_IPV6_PATTERNS`). Node's WHATWG URL parser canonicalizes `[::ffff:169.254.169.254]` to the **hex** form `[::ffff:a9fe:a9fe]`, which those regexes never matched — and because the host is an IP literal, the DNS-resolution fallback is skipped, so the miss was final. Confirmed live (Node 22): `[::ffff:169.254.169.254]` (cloud metadata), `[::ffff:127.0.0.1]`, `[::ffff:10.0.0.5]`, `[::ffff:192.168.1.1]`, and NAT64 `[64:ff9b::a9fe:a9fe]` all returned `isValid: true`. Reachable in-process via `capture_screenshot` (stdio) — an IP literal skips DNS-pinning, so `browserPage.goto` navigates to the internal target and captures it — and holes the defense-in-depth layer of every URL-forwarding tool.

**Fix:** `ssrf.ts` now expands any IPv6 literal to 16 bytes (`ipv6ToBytes`) and blocks loopback/link-local/ULA plus IPv4-mapped (`::ffff:0:0/96`), IPv4-compatible (`::/96`), and NAT64 (`64:ff9b::/96`) embeddings by extracting the embedded IPv4 and running it through the IPv4 blocklist (`isBlockedIPv6`). Genuine public IPv6 (`2606:4700:4700::1111`, mapped/NAT64 of public v4) still passes. 9 new regression cases in `ssrf.test.ts`.

> **Out-of-repo twin:** `ssrf.ts` is "Ported from `supabase/functions/_shared/ssrfProtection.ts`". If that copy shares the dotted-only IPv6 logic, the same bypass is live end-to-end on the hosted backend — apply the equivalent fix there.

### SSRF-3 — Generation tools forwarded `image_url`/`end_frame_url` unchecked · LOW→MED (defense-in-depth)
`generate_image` and `generate_video` forwarded a caller-supplied reference URL to `kie-image-generate`/`kie-video-generate` with no SSRF validation and no `.url()` — inconsistent with every sibling URL tool. The MCP process doesn't fetch these (the provider does), so exploitability depends on the backend, but the layer that guards all other URLs was simply absent.

**Fix:** both handlers now run a defense-in-depth SSRF check (`ssrfGuardImageUrl`) on any `http(s)` reference URL before dispatch, failing with a clean `… blocked: …` error and never reaching the provider EF. Non-http forms pass through as before. Regression test added.

### SIDE-EFFECTS-F4 — Pipeline credit deduction billed the wrong project · LOW (same-tenant mis-attribution)
`run_content_pipeline` passed explicit `resolvedProjectId` to every stage **except** the `deduct-credits` call (`pipeline.ts:523`), so `callEdgeFunction` auto-injected the caller's *default* project. A caller passing an explicit `project_id ≠ default` had generation credits debited from the wrong pool while the plan was created under the target project. Always within the caller's own account (identity is token-derived) — an attribution bug, not theft.

**Fix:** the deduct-credits call now passes `projectId`/`project_id: resolvedProjectId` for parity with every other stage.

## Confirmed live — fix lives in unmerged PR #249 (do NOT re-implement here)

### SIDE-EFFECTS-F1 — Recipe engine runs any effect class under only `mcp:write` · HIGH
`execute_recipe` on main is a thin pass-through (`slug` + `inputs` → `action:'execute-recipe'`), required scope `mcp:write` (`scopes.ts:108`), with **no** per-step effect→scope analysis, `project_id`, `dry_run`, or `confirm`. `analyzeRecipeEffects` does not exist in `src/` (grep hits only `docs/` and `scripts/`). A `mcp:write`-only key can therefore drive a recipe whose steps publish posts (`mcp:distribute`), moderate/delete comments (`mcp:comments`), or mutate autopilot (`mcp:autopilot`) — scopes it never held — and the gateway can't see per-step effects to stop it. Exposed in the default (`full`) tool profile.

This is exactly what **PR #249** ("closes recipe nested-scope escalation with exact effect classification, project binding, dry-run defaults, and live confirmation") fixes — its `recipes.ts` adds `analyzeRecipeEffects`, the union-of-scopes requirement, unknown-step default-deny, and `project_id`/`dry_run`/`confirm`. **Action: prioritize landing #249** (after the rebase + lock-regen from its review). Not fixed here to avoid conflicting with #249's larger rework.

### SCANNER-F3 — Output PII pass NFKC-normalizes then `JSON.parse`s the serialized result · MEDIUM (integrity/DoS)
`register-tools.ts:173-180` builds `sanitized_text` from the **normalized** JSON (`normalize()` applies NFKC + `\s{2,}→" "`) and then `JSON.parse`s it back into the returned object. Normalization isn't round-trip-safe over serialized JSON: an upstream 200 body with any redactable PII plus a fullwidth quote `＂`/backslash `＼` in a string value folds to a real `"`/`\` mid-string, so `JSON.parse` either throws (reliable DoS of any PII-bearing response → generic `server_error`) or restructures the object. This exact block is reworked by **PR #249** (per-leaf `scrubPii(normalize(value))` via a `JSON.stringify` replacer, which re-escapes correctly). **Action: land #249.** Reported, not double-patched.

## Reported for follow-up (verified, not fixed here)

Scanner/detector findings live in the `agent-harness` mirror, whose logic is owned by an upstream TS SSOT with a same-PR three-copy parity contract (`src/lib/agent-harness/README.md`) — they must be fixed in the SSOT, not patched in the mirror:

- **SCANNER-F1 · HIGH** — the instruction-phrase detector is a 10-pattern ASCII blocklist (`constants.json`). NFKC does **not** fold Cyrillic/Greek homoglyphs, so `"ignоre all previous instructions"` (Cyrillic `о`) passes both the raw and normalized scans and reaches the model verbatim. The `scanner.ts:47` comment claiming homoglyph coverage is false. Treat the scanner as defense-in-depth only; add Unicode confusable-skeleton folding and label upstream-derived output as untrusted data rather than relying on phrase detection.
- **SCANNER-F2 · MEDIUM** — output secret redaction is a format allowlist that **fails open**. AWS `AKIA…`, Google `AIza…`, Slack `xoxp/xapp`, `sno_` connector tokens (only `snk_` is in the regex), `Bearer <opaque>`, and PEM private-key blocks all reach the model unredacted. Add high-entropy/marker heuristics and redact-by-default on `mcp_tool_output`.
- **SCANNER-F4 · LOW/MED** — the zero-width class omits U+00AD (soft hyphen), U+2062–2064, U+FE00–FE0F, U+180E, U+061C; NFKC doesn't strip them. `"ig­nore…"` defeats the detector. Expand the class and strip default-ignorables in `normalize()`.

Rate-limit / OAuth / config hardening (deployment- or infra-shaped; recommend owning-team decides):

- **RATE-F2 · MEDIUM** — the limiter is in-memory per-process (`rate-limit.ts` / `http.ts` `ipBuckets`); on N Railway replicas the effective limit is N×, and the bucket key differs per surface (`mcp:` vs `rest:`), so a tool with no inline limiter (`execute_recipe`, `run_content_pipeline`, `generate_*`) gets the full category budget on each of `/mcp` and `/v1` → 2× per key. Not caller-forgeable (`userId` is token-derived). Move to a shared store keyed on `userId` with surface normalized out, or enforce the ceiling at the gateway.
- **AUTH-F2 · LOW→MED (misconfig)** — the `MCP_ALLOW_ANY_HTTPS_REDIRECT=true` DCR escape hatch is ignored only when `NODE_ENV==='production'`, but the app defaults `NODE_ENV` to `development` (`http.ts:76`) and never asserts it. A hosted deploy that sets the staging hatch but omits `NODE_ENV=production` would accept arbitrary HTTPS `redirect_uri`s → auth-code interception. Treat unset `NODE_ENV` as production for this guard, or refuse to boot on that combination. (OAuth-behavioral — left for the maintainer rather than shipped unilaterally.)
- **AUTH-F1 · LOW→MED** — the 11 `internal`-flagged tools (harness/hermes/loop-pulse/bandit) are hidden from discovery but gated only by baseline `mcp:read`/`mcp:write`; a low-tier key can invoke them by name (obscurity, not access control; still user-scoped). Introduce a dedicated non-public scope and require it in `TOOL_SCOPES`.
- **AUTH-F3 · LOW** — connector-token revocation evicts only the local process cache (60 s TTL); on multi-instance hosting a revoked `sno_` token stays valid up to 60 s on other instances. Add shared invalidation.
- **AUTH-F4 · LOW** — no local scope-subset assertion on refresh (`oauth-provider.ts:601-651`); a non-enforcing backend could widen a refreshed token (capped only to the 7 public scopes by `validScopes`). Add a defensive local subset check and a backend test proving the gateway rejects a foreign `project_id`.
- **AUTH-F6 · LOW (footgun)** — `SOCIALNEURON_PROJECT_ID` outranks DB resolution in `getDefaultProjectId` (`supabase.ts:163`); harmless on stdio, but a hosted process that read it would pin every unscoped caller to one project. Ignore it on the HTTP surface.
- **SSRF-2 · MEDIUM (stdio)** — `capture_screenshot` validates only the first URL; `goto(..., waitUntil:'networkidle')` follows 3xx redirects with no per-hop revalidation (`browser.ts` installs no `page.route`). A public URL that 302s to `169.254.169.254`/a private host is navigated and captured. Register a navigation/route guard that re-runs `validateUrlForSSRF` on every main-frame redirect target.
- **SIDE-EFFECTS-F5 · LOW→MED** — `schedule_content_plan` gates on approval **rows**, not `plan.status`: an inline `plan` is never approval-checked, and a persisted plan set to `in_review` but whose approval rows were never materialized schedules all posts ungated (`enforce_quality:false` also skips the quality gate). All-pending/all-rejected **is** correctly blocked. Load `plan.status` and refuse to schedule any non-approved plan.
- **SIDE-EFFECTS-F6 · LOW (consistency)** — `delete_comment` (permanent) and `moderate_comment` lack the `confirm: z.literal(true)` gate the `lifecycle.ts` delete/cancel tools enforce; `update_autopilot_config` can flip `is_active`/raise budget caps with no `confirm`/`project_id`. Add the confirm gate (schema change → tools.lock regen; hence flagged, not shipped mid-beta).
- **SIDE-EFFECTS-F3 · LOW→MED** — the local credit/asset budget (`budget.ts`) defaults to disabled, resets per request, and is referenced only by `content.ts`/`carousel.ts` — it is not a session-cumulative spend cap and doesn't cover recipes/pipelines/video. Treat the server-side balance as the real cap; if the local guard is defense-in-depth, make it session-cumulative and apply it in the engine paths.

## Controls verified to HOLD

Default-deny for unmapped tools; no scope-hierarchy climb; every tool wrapped (no `setRequestHandler` bypass); `mcp:internal` cannot be smuggled (`validScopes` filters to the 7 public scopes on all verify paths); session ownership enforced on POST/GET/DELETE `/mcp`; per-user + global session caps; OAuth resource binding fails closed; production fail-closed on missing resource; connected-account routing fails closed and is re-checked before spend; `getDefaultUserId`/`getDefaultProjectId` refuse to guess; upstream error sanitization (`safeGatewayError`/`safeFailureData`) never leaks bodies/URLs/tokens; the #307 exception wrapper includes only `err.name`; `redactBase64Payloads` mutates only the scan copy; output oversize + `JSON.parse`-fail both fail closed; telemetry sends only tool name/status/duration/error-class/HMAC'd userId (no inputs/outputs/keys); credential file I/O uses `O_NOFOLLOW`+`O_EXCL` with inode/uid/nlink checks and 0600/0700; the API key is never placed in a tool response (only prefix/suffix fragments on the operator's own stderr).
